> "In situations where a coroutine would be the natural implementation of a

Wow! Let me show you an example from my event driven http client.

This is the complicated state machine (called from [filevent $machine $client]):

proc ::htclient::statusMachine { client } {

    set state [getVar $client state]

    switch -exact -- $state {
	httpversion {
	    htHTTPVersion $client
	}
	statuscode {
	    htStatusCode $client
	}
	reasonphrase {
	    htReasonPhrase $client
	}
	cleanup {
	    log Debug "htCleanup $client"
	    htCleanup $client
	}
	default {
	    log Error "statusMachine: unknown state $state"
	    htError $client
	}
    }
}

There are multiple state machines to handle various tasks (sending a
request vs. receiving a response vs. receiving a chunked response).
The idea is that different machines have different states to handle
different circumstances. Machines can share state code since all
macines and all state procs share the same interface (they take the
clientID) as a single argument.

Note that every state machine also share the final two states:
"cleanup" and "default".

Looking through this code:

http://junom.com/gitweb/gitweb.perl?p=htclient.git;a=blob;f=htclient.tcl

I see exactly one use of [while], but it isn't used as a loop
construct. The only loop in the htclient code is provided by the event
loop. (There are a few [for] loops used to setup data). This is by
design: I needed a client that would work with a Tk application and
http::geturl switches channels to blocking mode when reading a chunk
encoded response...not good).

Anyway, I had thought I would need, or could use [yield] to handle a
difficult problem: how to keep large http requests (where the Tk
application uploads a large file) from freezing the application.

For instance here is my code for sending the request:

proc ::htclient::htSend { client } {

    set sock [getVar $client sock]

    puts -nonewline $sock [getVar $client request]

    flush $sock

    if {![htPending output $sock]} {
	log Debug "htSend sent '[getVar $client request]'"
	setVar $client state httpversion
    } else {
	log Debug "htSend still sending pending = [htPending input $sock]"
	setVar $client state sendwait
    }

    log Debug "htSend end state $client = [getVar $client state]"
}

I know those first three lines are complicated, and the following [if]
can be removed, since [chan pending] will always return 0 after [puts]
returns. The log statement is obviously not part of the control
structure, so we basically have a total of four lines of effective
code here...very complicated stuff.

Now here is what I did so that a large $request wouldn't take up too much time:

proc ::htclient::htSendChunk { client } {

    variable clientSendBufIndex

    set chunkSize 20
    set requestLength [getVar $client request-length]

    set lastChunk 0

    if {$clientSendBufIndex($client) + $chunkSize <= $requestLength} {
	set chunkEnd [expr {$clientSendBufIndex($client) + $chunkSize - 1}]
	set chunk [string range [getVar $client request]
$clientSendBufIndex($client) $chunkEnd]
	set clientSendBufIndex($client) [expr {$chunkEnd +1}]
    } else {
	set chunkEnd end
	set chunk [string range [getVar $client request]
$clientSendBufIndex($client) $chunkEnd]
	set clientSendBufIndex($client) $requestLength
	set lastChunk 1
    }

    set sock [getVar $client sock]

    puts -nonewline $sock $chunk

    flush $sock

    if {$lastChunk} {
	if {![htPending output $sock]} {
	    log Debug "htSendChunk sent '[getVar $client request]'"
	    setVar $client state httpversion
	} else {
	    log Debug "htSendChunk still sending pending = [htPending input $sock]"
	    setVar $client state sendwait
	}
    }

    log Debug "htSendChunk end state $client = [getVar $client state]
chunkEnd = $chunkEnd"
    log Debug "htSendChunk sent: '$chunk'"
}

Basically just a little more complicated. I hard coded the "chunk"
size for testing 20 chars allows me to use small requests for testing
and see more than one invocation of the above code per request.

The new code just maintains an index into the $request and sends a
certain length chunk from this string. The last send, changes the
state.

I thought about this for a few hours before I realized that [yield]
would not help at all here.

I ran the head of the above code on a few urls, the code and log files are here:

What you see in the log files is five simultaneous requests/responses
(geturl does one at a time, no overlap). I used wish8.6 to source the
script file. You could write a Tk GUI which would allow you to click
on a current request and cancel it (by setting the state to "cleanup",
or executing htCleanup if no events are firing). htCleanup only
deletes state data and closes the socket, request/response still
exist. another procedure htEraseClient is used to free all resources.

This type of design would make much more sense with a POP
client/server since the conversation is considerably more complicated.

Please don't get the idea that I'm an event cheerleader: I use
AOLserver which uses threads, but I wrote htclient in an attempt to
understand events in Tcl.

Now I'm going to work on using coroutines to stop/start long running
computations (using a sudoku solver as the example ... the time to
solve some puzzles is many orders of magnitude higher than the typical
puzzle).


> In the view of various other people, dividing sequential logic into separate
> "sub-events" is a work-around for not having coroutines, and is harder to
> maintain.

It's called modular programming. The new NRE core follows this concept
(a little) by dividing a single command implementation into three
potentially reusable C functions.

> A more modular approach may involve an object storing the state, and the
> logic being split into different methods.  The effect is the same: if you
> don't have coroutines, you may have to refactor your code to use events.

You can't handle I/O in Tcl without events, my htclient code proves
(it is still example code, but is large enough to demonstrate
capabilities) you don't need coroutines or complicated code to use

That doesn't make coroutines any less interesting.

Re: [TCLCORE] TIP #373: Improved Yielding Support for Coroutines
From: Donal K. Fellows <donal.fellows@ma...> - 2010-08-20 15:42:32
Attachments: donal_k_fellows.vcf     
On 20/08/2010 16:12, Tom Jackson wrote:
> So basically you run an additional coroutine tied to the original
> caller? And returning from this proc skips by the first coroutine
> (which called [yieldto ...]) and returns to the caller of $c.

I'll try to do a bit of summarization. I don't plan to explain 
everything though; there's a lot of background material elsewhere. 
(Notably the Wiki and archives of this list from 2008.)


Current status:

There's at most one current coroutine per interpreter. (If there was 
always exactly one, we'd have a fully symmetric coroutine system AIUI, 
but we don't.)

[yieldto] stops the current coroutine and, in effect, tailcalls 
something else (except that unlike with [tailcall], you can resume the 
coroutine to get back to where you were). When the coroutine is resumed, 
the optional value passed in is the result of [yieldto]. The other thing 
called could be a resumption of another coroutine, but need not be.

[yield] stops the current coroutine and returns a value to the 
caller/resumer of the coroutine. When the coroutine is resumed, the 
optional value passed in is the result of [yield].

Every coroutine has a name (returned by [info coroutine]) which is the 
name of the command that is used to resume the coroutine. Deleting that 
command deletes the coroutine (running any unset traces defined for 
variables on stack frames that are local to the coro).

The resumption command takes a single, optional value that forms the 
result of the [yield]/[yieldto] inside the coroutine.

Resuming a coroutine inside another coroutine, unlike yielding to it, 
keeps the "outer" coroutine around, waiting for the result/yielded value.

There is an unsupported [yieldm] which is just like [yield] except it 
makes the coroutine take arbitrarily many resumption arguments; the 
[yieldm] returns the list of these values.


Proposals on the table:

Expose [yieldto] (currently unexposed). Not very controversial since it 
appears to be useful and unduplicated (as well as being able to emulate 
some neat useful things). Note that this can pass arbitrarily many 
arguments to the command that is yielded to.

Expose something that makes [yield] able to resume with multiple 
arguments being passed in, in effect exposing [yieldm]'s capabilities. 
Colin McCormack asserts that this is useful and is known to have an 
application of a stature able to back this up with in-service 
experience, and it is certainly nicer than having to manually pack 
things up with [list] before passing them in. (Just because we can do 
things by packing doesn't mean that we should; it's just making things 
harder than necessary.) The [yieldset] command was a proposal for doing 
this, but arguably it is better to use some optional arguments to 
[yield] instead. [yieldto] would also need some alteration if [yield] 
was modified that way.

If [yield] and [yieldto] have any option arguments, arguably [yield] 
should support all [return] options (just as if they were [yieldto 
return]). This is less than critical; just a tidying up.


Other notes:

Some people in this discussion are presenting very low-value arguments. 
A reminder to all: this is a technical group, so only **technical** 
arguments are valid and persuasive. One who attempts to be dismissive of 
others is truly just cheapening their own position.

Donal.

One that comes to mind is implementing a POP server.

IIRC, messages to the server all take the simple form "command 
argument", so a coroutine acting as "the server" (more properly: 
handling one connection to it) would be naturally implemented by a 
coroutine with

   yieldset {cmd arg}

(I think later extensions to the protocol adds commands with more than 
one argument, so one might make that [yieldset {cmd arg args}] just to 
be on the safe side, but that's a minor detail.) This makes 
experiments-at-tclsh-prompt with the server blazingly obvious, as you 
can merely "connect to it" by

   coroutine test pop::serve_connection ...possibly-more-args...

and enter your commands as

   test USER avl
   test PASS swordfish

The two main states of the connection would be "not logged in" and 
"logged in", so a basic structure of pop::serve_connection could be

proc pop::serve_connection {} {
    set logged_in 0
    while {!$logged_in} {
       # Authentication processing
    }
    while {$logged_in} {
       # Serve commands for particular user
    }
    return "Bye"
}

This assumes logging out implies closing the connection; I don't recall 
if that's the case (and the bye message probably didn't look quite like 
that), but it's no big deal to fix.

One point where doing this as a coroutine really simplifies the code is 
in the login sequence, since the user is first supposed to send a USER 
command and then immediately after a PASS command. With yielding, that 
is merely

  set res [initial_greeting]
  while {!$logged_in} {
     yield -args {cmd arg args} $res
     switch -- $cmd USER {
        yield -args {PASS password} "nnn Enter PASS"
        if {$PASS ne "PASS"} then {
           set res "nnn Wrong command, must be PASS"
        } elseif {[info exists Passwd($arg)] &&\
                  $Passwd($arg) eq $password {
           set logged_in 1
           set user $arg
        } else {
           set res "nnn Wrong password, or user unknown"
        }
     }
  }
  set res "nnn Welcome $user, you have [unread $user] new messages"

In a non-coroutine implementation, it would instead be necessary to 
handle this quirk in the protocol by adding an extra state to the machine.

Hope that helps,

No.  "yieldto implements 'suspend yourself and tailcall the new 
command'".  $c can be any command, including for example [return] 
(allowing errors to be returned to the coroutine caller, as described in 
the TIP).

[yieldset] modifies the behaviour of the running coroutine, defining the 
arglist of its resume command.

Example:

   proc puts_and_return {val} {
     puts "output: $val"
     return $val
   }

   proc accum {a b} {
     yieldset {b x y}
     set total $a
     while { $b ne {} } {
       lassign [yieldto puts_and_return [incr total $b]] b x y
       # x, y unused
     }
   }

   coroutine coro accum 1 2 ;# -> 3 (and puts "output: 3" to console)
   coro 4  ;# -> wrong # args, should be "coro b x y"
   coro 5 foo bar ;# -> 8 (and puts "output: 8" to console)
   coro -1 a b ;# -> 7 (and puts "output: 7" to console)

* [coroutine] executes "[uplevel #0 [list accum 1 2]]" (such that its 
stack can be suspended)
* [yieldset] makes a note of the arglist for $coroCmdName
* [yieldto] suspends the execution of [accum], creates a command 
$coroCmdName (in this case "coro") that accepts the arglist noted by 
[yieldset], then [tailcall]s "puts_and_return ...".
* [puts_and_return] runs and returns to the coroutine caller.
* Later we invoke [coro] to resume the coroutine.

On 8/19/2010 3:51 PM, Tom Jackson wrote:
> In summary: there are two situations: yielding because a particular
> computation takes too long (or never completes), or yielding because
> the developer doesn't want to refactor their event code. Choose
> wisely.

Wibble uses coroutines and fileevent together in a valid and gainful 
way.  It takes advantage of the fact that Tcl coroutines are stackful, 
meaning that [yield] works even when it's buried deep in the stack. 
This allows the code that services each connection to be written simply, 
in the style of straight-line blocking I/O code.  But behind the scenes, 
the "blocking I/O" uses [yield] so that it only blocks the current 
coroutine, yet allows coroutines for other connections to be started or 
resumed by [socket -server] or [fileevent], respectively.  It's very 
difficult to rewrite such stackful code to use explicit continuation 
tracking, and the result of this process is code that's harder to 
understand due to having been split up anywhere a [yield] could happen.

You see, Wibble's coroutine is truly a long-running routine, not merely 
an event handler.  Well, maybe it is an event handler, if the event is 
"I got a connection" and the handler is "process all I/O for the 
connection until it closes, even in the face of pipelining and 
keepalive".  That's a large event, a super-event as you say, requiring a 
large handler.  Therefore it's better to think of it as a routine.  But 
multiple connections have to be serviced simultaneously, so the routine 
has to be a coroutine.

Instead, the event is simply "incoming data is available", and the 
handler is simply "resume the coroutine".  Let me underscore:  The 
coroutine is NOT the event handler, the event handler is NOT the 
coroutine.  The event handler is: RESUME the coroutine.  That's 
literally what it is.  [fileevent $chan readable [list [info 
coroutine]]] says: when $chan is readable, resume the current coroutine. 
  You will note that the coroutine already exists, before there ever was 
a readable event.

Think about what happens when you go to bed for the night, catch forty 
winks, then wake up when the alarm clock rings.  Your life is a 
coroutine, and going to sleep is yielding.  What's the handler for the 
alarm-clock-ring event?  Is it to get out of bed, brush your teeth, put 
on pants, and go to work?  No, all those steps are what's already 
written into the coroutine of your life.  The event handler is to simply 
[wake you up] so that [you will move on] to the next step of your life. 
  It's a fine point; the computer can't tell the difference.  But such 
philosophical distinctions can shift your perspective and enable you to 
grasp the design of this admittedly complex system.

Isn't this pure nominalism?  By giving something a name, one presumes 
that by that act alone it shares properties with other things which 
happen also to share that name?

Let's see if I can sketch it:  there are things called factory commands, 
they are things which create other commands.  Nothing in the definition 
of factory command entails an arity of the commands produced.  Let 
[coroutine] to be a factory command.  (now watch what I do here) Because 
factory commands entail nothing of the arity of their product, it 
follows that coroutine is free to implement any arity it wishes.

It's argument by analogy, fragile at best.

Try this:  There is a category of command called Tcl command, and a 
subset of those produce commands.  All of the existing instances of that 
category, but [coroutine], produce variadic commands.  The creation of a 
Tcl command producing a command does not logically entail that the 
command so produced must be variadic, but the general principle 'it is 
best if things which do similar things do them in similar ways' suggests 
that it ought.

Or, if we tire of sterile scholasticism, how about an empirical 
argument:  My code is cleaner and shorter if coroutines are variadic.  
It is reasonable to suppose that others' code also might benefit.
